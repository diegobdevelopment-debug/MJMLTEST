# MJML Email Templates

 Templates are authored in MJML and compiled to plain HTML at build time. The compiled HTML files are what the application actually sends.

## Folder structure

```
mjml/
├── templates/        # One .mjml file per email
│   └── welcome.mjml
├── partials/
│   ├── styles/       # CSS-only, included in <mj-head>
│   │   ├── base.mjml       # Link colours, hover states
│   │   └── dark-mode.mjml  # Dark mode overrides + color-scheme meta tags
│   └── components/   # Layout fragments, included in <mj-body>
│       ├── header.mjml
│       └── footer.mjml
├── output/           # Compiled HTML (gitignored, generated by build)
├── scripts/
│   ├── build.js      # Compiles all templates (or a single one)
│   ├── watch.js      # Re-compiles on file changes
│   ├── render.js     # Runtime helper: loads compiled HTML + fills placeholders
│   └── send.js       # Dev tool: renders + sends a template via Mailtrap
├── .env              # Local credentials (gitignored)
├── .env.example      # Checked-in template for .env
└── package.json
```

## Getting started

```bash
npm install
npm run build        # compile all templates → output/
npm run watch        # recompile on save (for local development)
```

Build a single template:

```bash
npm run build:template -- welcome
```

## Authoring templates

- Add a new file under `templates/` (e.g. `templates/order-confirmation.mjml`).
- Include styles from `partials/styles/` inside `<mj-head>`, layout components from `partials/components/` inside `<mj-body>`.
- Use `{{placeholderName}}` syntax for dynamic content (name, URLs, etc.). These are replaced at runtime by `scripts/render.js`.
- Run `npm run watch` while editing so output refreshes automatically.

MJML docs: https://documentation.mjml.io

## Using compiled templates in the app

Copy or require `scripts/render.js` into your Node.js project and point it at the `output/` directory.

```js
const render = require('./mjml/scripts/render');

const html = render('welcome', {
  name: 'Alice',
  ctaUrl: 'https://example.com/start',
  unsubscribeUrl: 'https://example.com/unsubscribe',
});

// Pass `html` to your email sending library (nodemailer, SES, etc.)
```

`render()` throws a descriptive error if the compiled file is missing, so a failed build is caught before sending.

## Placeholders

Document the variables each template expects:

| Template  | Variables                          |
|-----------|------------------------------------|
| welcome   | `name`, `ctaUrl`, `unsubscribeUrl` |

## Testing with Mailtrap

Emails are sent to [Mailtrap's sandbox](https://mailtrap.io) during development — nothing reaches a real inbox.

### Setup

Copy `.env.example` to `.env` and fill in your Mailtrap sandbox credentials (found under **Email Testing → Inboxes → SMTP Settings** in the Mailtrap UI):

```bash
cp .env.example .env
```

`.env` is gitignored. Never commit real credentials.

### Send a test email

```bash
# build first so output/ is fresh
npm run build

# send with placeholder values
npm run send -- welcome alice@example.com '{"name":"Alice","ctaUrl":"https://example.com","unsubscribeUrl":"https://example.com/unsub"}'
```

Check the result in your [Mailtrap inbox](https://mailtrap.io/inboxes). The subject line is prefixed with `[test]` so sandbox sends are easy to spot.

### Using the transport in the existing Node.js app

`scripts/send.js` is a standalone test tool. For the actual app, create the nodemailer transport using the same env vars:

```js
const nodemailer = require('nodemailer');

const transport = nodemailer.createTransport({
  host: process.env.MAILTRAP_HOST,
  port: Number(process.env.MAILTRAP_PORT),
  auth: {
    user: process.env.MAILTRAP_USER,
    pass: process.env.MAILTRAP_PASS,
  },
});
```

Swap in your production SMTP credentials (SES, SendGrid, Postmark, etc.) by changing those four env vars — the rest of the code stays the same.

## Dark mode

Email clients handle dark mode in three different ways, so the setup covers all three simultaneously.

| Layer | Clients covered |
|---|---|
| `@media (prefers-color-scheme: dark)` | Apple Mail, iOS Mail, Gmail app, Samsung Mail, Thunderbird, Yahoo Mail |
| `[data-ogsc]` attribute prefix | Outlook.com, Office 365 web |
| `color-scheme` meta tags | Tells clients you've designed dark mode yourself, suppressing their auto-inversion |

### How to apply dark mode to a new template

1. Include both partials in `<mj-head>`:

```xml
<mj-include path="../partials/head-styles.mjml" />
<mj-include path="../partials/head-dark-mode.mjml" />
```

2. Add `css-class` attributes to your sections, columns, and text elements using the predefined tokens:

| Class | Overrides |
|---|---|
| `dm-body` | Outer page background (`mj-body`) |
| `dm-card` | White content card (`mj-section`) |
| `dm-hero` | Brand-colour header (`mj-section`) |
| `dm-footer` | Footer background (`mj-section`) |
| `dm-heading` | Large heading text (`mj-text`) |
| `dm-text` | Body text (`mj-text`) |
| `dm-muted` | Secondary/small text (`mj-text`) |
| `dm-btn` | Button wrapper (`mj-button`) |

```xml
<mj-body background-color="#f4f4f4" css-class="dm-body">
  <mj-section background-color="#ffffff" css-class="dm-card">
    <mj-column>
      <mj-text color="#111111" css-class="dm-heading">Hello!</mj-text>
      <mj-text css-class="dm-text">Body copy here.</mj-text>
      <mj-button background-color="#0066cc" css-class="dm-btn" href="…">
        Click me
      </mj-button>
    </mj-column>
  </mj-section>
</mj-body>
```

3. All dark colour tokens are defined in `partials/head-dark-mode.mjml`. Edit them there to update every template at once.

### Swapping images for dark mode

Add `css-class="light-img"` or `css-class="dark-img"` to `mj-image` elements. The CSS hides/shows them based on the active scheme:

```xml
<mj-image src="logo-light.png" css-class="light-img" … />
<mj-image src="logo-dark.png"  css-class="dark-img"  … />
```

### Forcing light mode

If you want a template to always render in light mode (ignoring the user's OS setting), change the meta tag in `head-dark-mode.mjml`:

```xml
<meta name="color-scheme" content="light only" />
<meta name="supported-color-schemes" content="light only" />
```

## CI / build pipeline

Run `npm run build` as part of your build step to ensure `output/` is always fresh before deployment. Since `output/` is gitignored, compiled HTML should be generated in CI rather than committed.
